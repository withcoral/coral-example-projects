# Spike: opencode as an alternative agent backend

Status: research spike (no code changes). Evaluates whether `opencode` could
replace the Pydantic AI investigation loop in `SRE-agent/`.

## Current baseline (what we are comparing against)

`SRE-agent/src/sre_agent/agent.py` — `PydanticSreAgent` wraps Claude via
[Pydantic AI](https://ai.pydantic.dev/). It builds an `Agent` with the Coral MCP
server attached as an `MCPServerStdio` toolset, runs a bounded multi-round
tool-call loop (`UsageLimits`), and returns an evidence-backed text answer. The
whole thing is an in-process Python library call inside a long-running Slack bot.

## What opencode is

`opencode` (styled OpenCode) is an open-source **AI coding agent built for the
terminal** — a direct competitor to Claude Code / Cursor / Copilot. It is
maintained by Anomaly Innovations (`anomalyco`, formerly `sst/opencode`), MIT
licensed, and very active (160k+ GitHub stars, 900+ contributors). It ships as a
terminal TUI, desktop app, and IDE extensions, with a strong "local-first,
privacy-first, no vendor lock-in" positioning.

Key fact for us: it is **primarily designed as an interactive coding assistant**,
not as an embeddable investigation/agent runtime. But it does have a real
programmatic surface (see below), so embedding is not off the table.

## How integration would actually work

opencode has a client/server split. `opencode serve [--port]` runs a **headless
HTTP server** (OpenAPI 3.1, default `127.0.0.1:4096`) — no TUI required. The TUI
is just one client of that server. Our Python bot would talk to that server.

- **Python SDK exists but is pre-release.** `opencode-ai` on PyPI
  (`anomalyco/opencode-sdk-python`, MIT, Python 3.8+, sync + async, generated by
  Stainless) is a typed REST client for the server. It must be installed with
  `pip install --pre` — i.e. **not yet 1.0 / not API-stable**.
- **Integration shape:** the bot would (a) spawn/manage an `opencode serve`
  subprocess, then (b) use the SDK to `session.create()` and
  `session.prompt(...)`, streaming events via `event.subscribe()`. This replaces
  the in-process `Agent.run()` call with an out-of-process server + HTTP client —
  an extra moving part to supervise inside the Slack bot.
- **Workspace assumption:** opencode operates within a project directory
  (VCS endpoints, file search, LSP). For a read-only SRE bot with no code
  workspace this is dead weight, though it can run against an empty/throwaway dir.

### MCP and tool use

This part fits well. opencode is a first-class MCP client: MCP servers are
declared in `opencode.json` under the `mcp` key, supporting both `local`
(stdio, command array) and `remote` servers. Coral exposes a stdio MCP server
(`coral mcp` — see `coral_mcp.py`), so we could register Coral as a `local` MCP
server and opencode's agent loop would call the `sql` / `list_tables` tools the
same way Pydantic AI does today. opencode also has built-in coding tools and an
`Allow/Ask/Deny` permission model, plus a read-only `plan` agent that maps
nicely onto our read-only SRE constraint.

### Model support

Strong. opencode supports 75+ providers including Anthropic Claude, OpenAI,
Gemini, and local models via Ollama. Running `claude-sonnet-4-6` is supported,
so no regression on the model side, and more provider flexibility than today.

## Pros / cons vs Pydantic AI for this use case

| Dimension | Pydantic AI (current) | opencode |
| --- | --- | --- |
| Integration model | In-process Python library | Out-of-process HTTP server + SDK client |
| Python SDK maturity | Stable, widely used | Pre-release (`--pre`), API not frozen |
| MCP / Coral support | Native stdio toolset | Native (`mcp` config, local stdio) |
| Model support | Anthropic + others | 75+ providers incl. Claude |
| Loop control | Direct (`UsageLimits`, rounds) | Indirect, via server session/config |
| Fit for purpose | Built as an agent framework | Built as a *coding* agent / TUI |
| Operational weight | Just an `await` in the bot | Must supervise a serve subprocess |
| License | OSS | MIT |
| Cost | Model API only | Model API only (tool itself free) |

**Pros of switching:** broader provider flexibility, MIT, built-in
permission/plan modes, an actively developed ecosystem.

**Cons:** opencode is solving a different problem (interactive coding in a repo).
For our use case it adds an HTTP server to babysit, ships an unstable Python SDK,
assumes a code workspace we don't have, and hands us *less* direct control over
the bounded tool-call loop that is the core of an SRE investigation. None of
those are blockers, but they are pure cost with little upside — the one thing we
gain (provider flexibility) Pydantic AI also offers.

## Recommendation

**Do not adopt opencode as the agent backend. Keep Pydantic AI.**

Pydantic AI already gives us exactly what this loop needs: an in-process,
stable, well-controlled multi-round tool-call agent with native MCP and
multi-provider support. opencode is an excellent *coding agent* but is mismatched
to an embedded, read-only, workspace-less SRE investigation loop, and its Python
SDK is still pre-1.0. Revisit only if (a) we need opencode-specific features
(permission UX, its agent presets) or (b) the Python SDK reaches a stable
release and we have a concrete reason to move out-of-process.
